# EventRelay — AWS S3 Storage and Policies

This document details the S3 bucket configuration, access policies, and lifecycle rules implemented in EventRelay to handle historical payload archiving and logs storage.

---

## 1. Storage Buckets Directory

EventRelay provisions three private S3 buckets:

| Bucket Name | Purpose | KMS Key Encryption | Versioning |
|-------------|---------|-------------------|------------|
| `eventrelay-historical-backups` | Monthly database partition exports. | `aws/s3` | Enabled |
| `eventrelay-payload-overflow` | Out-of-band storage for webhooks $> 256\text{ KB}$. | `aws/s3` | Disabled |
| `eventrelay-alb-logs` | HTTP access logs from ALB. | `aws/s3` | Disabled |

---

## 2. S3 Lifecycle Policies

To balance compliance storage requirements with AWS costs, lifecycle rules are configured on the archive bucket:

```
[ Ingestion / Export ] ──► [ S3 Standard (0-30 Days) ] ──► [ S3 Infrequent Access (30-90 Days) ]
                                                                      │
                                                                      ▼
                                                              [ AWS Glacier ]
                                                              (Retained: 7 Years)
```

- **Standard Storage**: Partition exports land in standard S3 storage.
- **Transition to Infrequent Access (IA)**: After 30 days, objects transition to IA (saving $40\%$ storage costs).
- **Transition to Glacier**: After 90 days, objects transition to Glacier Flexible Retrieval for long-term archiving.
- **Expiration**: After 7 years (2,555 days), files are permanently deleted.

## 3. Bucket Policy Example (Historical Backups)

This bucket policy restricts access to the ECS task execution role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowECSAccessOnly",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/eventrelay-dispatcher-task-role"
      },
      "Action": [
        "s3:PutObject",
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::eventrelay-historical-backups/*"
    },
    {
      "Sid": "DenyUnencryptedUploads",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::eventrelay-historical-backups/*",
      "Condition": {
        "Null": {
          "s3:x-amz-server-side-encryption": "true"
        }
      }
    }
  ]
}
```

## 4. Payload Overflow Pattern

While SQS limits messages to 256KB, EventRelay supports up to 10MB payloads.
- If a client posts a payload $> 256\text{ KB}$, the Ingest API uploads it to `s3://eventrelay-payload-overflow/{event_id}`, and places a reference SQS message containing the S3 URL in the queue.
- The Dispatcher Worker detects the S3 reference, fetches the payload from S3, executes the HTTP call, and deletes the transient S3 object.
